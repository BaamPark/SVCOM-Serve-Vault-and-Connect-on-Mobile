# Tool Emoji Positioning — Problem & Solution

## Problem

The floating tools button (🛠) on the note page had unstable positioning on iPhone when the keyboard opened.

- When the button was in a **top corner**, opening the keyboard pushed it off-screen above the visible area.
- When the button was in a **bottom corner**, opening the keyboard left it hidden behind the keyboard.
- Attempts to reposition it by swapping bottom corners to top corners caused jarring jumps.
- When cursor-tracking was added, the button drifted upward while typing.
- When the cursor was tapped **below the keyboard area**, the button was misaligned with the cursor after iOS scrolled the page to reveal it.

---

## Root Cause

Two coordinate system mismatches on iOS Safari:

### 1. `getBoundingClientRect()` vs `position: fixed`

- `getBoundingClientRect()` returns coordinates in the **visual viewport** coordinate system.
- `position: fixed` places elements in the **layout viewport (ICB)** coordinate system.
- When the keyboard opens, iOS shifts the visual viewport relative to the layout viewport. This offset is exposed as `window.visualViewport.offsetTop`.
- If `vvp.offsetTop = 219px`, a cursor at `rect.top = 388px` (visual coords) is actually at `388 + 219 = 607px` in layout coords. Setting `top: 388` on a fixed element misses the cursor by 219px.

### 2. Wrong scroll event

- iOS auto-scrolls the page when the keyboard opens to reveal the cursor. This changes `window.scrollY`.
- We were listening to `visualViewport.scroll` to detect this, but iOS page scroll fires `window.scroll` — not `visualViewport.scroll`.
- So cursor position was being re-read before iOS finished scrolling.

---

## Debugging

Added an on-screen debug overlay (visible on iPhone) that logged on keyboard open:

```
cursor on tap:   n/a
cursor after:    388.6px
computed top:    361.0px
scrollY:         911 → 1130px
scroll delta:    219.0px ↑ up (page scrolled down)
vvp.height:      417.0px
vvp.offsetTop:   219.3px
innerHeight:     797px
```

The key observation: `scroll delta (219.0) ≈ vvp.offsetTop (219.3)`.
This confirmed that `vvp.offsetTop` is the exact coordinate gap between the two systems.

---

## Solution

### Cursor alignment

In `updateCursorAlignedY`, add `vvp.offsetTop` when converting cursor rect to fixed position:

```js
const offsetTop = vvp?.offsetTop || 0;
const cursorCenter = rect.top + offsetTop + rect.height / 2;
const buttonTop = cursorCenter - floatingButtonSize / 2;
setCursorAlignedY(
  Math.max(offsetTop + margin, Math.min(buttonTop, offsetTop + vvp.height - buttonSize - 4))
);
```

### Corner positions

In `floatingCornerPosition` and `clampFloatingDragPosition`, use `vvp.offsetTop` as the top boundary when keyboard is open:

```js
const offsetTop = keyboardInset > 0 ? (vvp?.offsetTop || 0) : 0;
const top = corner.startsWith("top")
  ? offsetTop + margin
  : offsetTop + effectiveHeight - bottomOffset - buttonSize;
```

### Scroll event

Changed from `visualViewport.scroll` to `window.scroll` to correctly detect iOS auto-scroll:

```js
window.addEventListener("scroll", handleWindowScroll, { passive: true });
```

After scroll settles (60ms debounce), re-read cursor position with the correct coordinates.

### Cursor tracking behavior

- Button tracks cursor position only on **pointer up** (tap), not on every `selectionchange` (which fires on every keystroke and caused drift while typing).
- On keyboard open, position is read after a 120ms delay, then corrected again once `window.scroll` settles.

---

## Final UI

- Removed sticky header toolbar and fixed bottom toolbar.
- The 🛠 floating button is the sole tool surface.
- When keyboard is closed: button sits in its corner.
- When keyboard opens: button moves to the cursor's vertical level, staying on the preferred left/right side.
